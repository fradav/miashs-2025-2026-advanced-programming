And nowâ¦ parallelize

Question : is pattern matching like we just did an âembarrassingly parallelâ problemâ¯?â¯Explain.

It shouldnât be : if we partition the data in chunks, the pattern matching will missâ¯any match occuringâ¯between two consecutive chunks.

Numba got a powerful (multi-threaded) parallelization feature, one justâ¯needs to : 1. add parallel=Trueâ¯in the decoratorâ¯call 2. replace pythonâ¯rangeâ¯used for looping with numbaâs prange.

With a spetial attention to whereâ¯you could put parallelization directive with prange (remember the âConceptsâ course). Test and benchmark, give the speedup and comment.

Whyâ¯there si no race conditionâ¯thereâ¯? (Tip : consider concurrent access in multi-threading, and look closely in the loop to read/store to the data).

Code 
import numba

@numba.jit(nopython=True,parallel=True)
def search_sequence_numba_parallel(data,seq):
    cor_size = data.size-seq.size+1
    
    matches = np.ones(cor_size,dtype=np.uint8)

    for i in numba.prange(cor_size): # walking on data
        for j in range(seq.size): # walking on sequence
            if data[i+j] != seq[j]:
                matches[i] = 0      
                break
                
    return np.nonzero(matches)[0]

import multiprocessing
from itertools import chain

def search_sequence_multiprocessing(data,seq,ncores):
    cor_size = data.size-seq.size+1
    
    mapargs = [(data,seq,chunk) for chunk in chunks(np.arange(cor_size),int(cor_size/ncores))]

        res = p.starmap(search_sequence_numba2,mapargs)
    matches_index = list(chain(*res))
